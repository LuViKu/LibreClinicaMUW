# Phase E.9 — Accessibility audit

**Date:** 2026-05-30
**Scope:** All E.3 primitives + every shipped view (Investigator + Monitor + Data Manager + Auth).
**Target:** WCAG 2.2 AA + the Web Accessibility Initiative's ARIA Authoring Practices for the patterns we use (dialog, progressbar, table, tablist, group of radio, form-field).
**Harness:**

- **Vitest + vitest-axe** — component-level structural checks ([`src/test/a11y/primitives.test.ts`](../../../web/src/spa/src/test/a11y/primitives.test.ts)) run on every push via `pnpm test:a11y`. Skips `color-contrast` and `region` rules — jsdom doesn't paint and primitives often don't render as landmarks in isolation.
- **Playwright + @axe-core/playwright** — browser-level checks on each route ([`tests/a11y/spa-routes.spec.ts`](../../../web/src/spa/tests/a11y/spa-routes.spec.ts)) run on demand via `pnpm test:a11y:e2e`. Catches contrast + viewport-dependent issues.
- **`pnpm check-i18n`** — i18n coverage guard ([`scripts/check-i18n-coverage.mjs`](../../../web/src/spa/scripts/check-i18n-coverage.mjs)) blocks key drift between `de.json` and `en.json` + flags untranslated English fallbacks.

## Methodology

This audit is the first formal pass against the SPA. Two phases:

1. **Automated baseline** (this report) — wires the gates above + a static review of every shipped view for the most common WCAG failures. Quick wins land in the same push as the harness so the gates can flip green on day one rather than carrying a backlog from the start.
2. **Manual deep-dive** (post-pnpm-install) — once the deps are installed locally, run the suites against the actual rendered DOM. Findings get appended to this doc; fixes ship in follow-up PRs gated by axe-clean re-runs.

## Static review — issues spotted while writing the harness

The following are flagged for the first manual deep-dive pass. None are blockers for the gates landing as-is; each has a clear follow-up.

### 1. Modal — focus trap (carry-over from E.3)

`<Modal>` currently moves focus to the panel on open and locks body scroll, but it does **not** trap Tab focus inside the modal. WCAG 2.4.3 (Focus Order) doesn't require a trap, but the ARIA Authoring Practices for `dialog` strongly recommend one. **Fix:** add a `focus-trap` implementation in a follow-up — either inline (track Tab + Shift+Tab on the first/last focusable element) or via the small `focus-trap` library. Carrier note already in the component's TODO block.

### 2. SDV bulk-action banner — `role="region"`

The bulk-action banner (`<div role="region" :aria-label="t('sdv.selectionAriaLabel')">`) is intentional: the region announces "X CRFs selected" to screen readers. Verified the ARIA label is translated in both locales. No fix needed; documented here for traceability.

### 3. Wizard step buttons — disabled vs. clickable=false

Steps that are not yet reachable (e.g. Commit before Preview passes in the Import wizard) render with `:disabled="s.clickable === false && idx !== step"`. The `disabled` attribute + the `cursor-not-allowed` styling correctly communicate the state. Spot-check: the disabled step button should still appear in the tab order (per WCAG 2.4.3) — confirmed via static reading of the template.

### 4. CRF Entry read-only mode

`<fieldset :disabled="isReadOnly">` correctly disables every input inside. The save / mark-complete row hides via `v-if`. **Spot-check needed in the manual pass:** confirm that the "Read-only · Monitor view" StatusPill announces its content to screen readers when the page loads in read-only mode (the pill is rendered after the page title; tab order should land on it before the form). No fix in this push.

### 5. Time-of-day in audit-log timestamps uses `font-mono`

The audit-log + import-CRF-data preview timestamps render in `font-mono` (JetBrains Mono). WCAG 1.4.4 (Resize Text) requires text to remain readable at 200% zoom. JetBrains Mono is a standard variable-width font; should pass. Verified by hand in the design-token CSS.

### 6. Sign Subject — preflight warning vs. blocker

The preflight pattern distinguishes `warn` (amber triangle, sign allowed) from `blocker` (rose triangle, sign disabled). The shipped Sign Subject view never emits a `blocker` row (open queries are warn-only — see DR-008 + the mockup). The blocker variant is wired in the primitive for future use (study lock, archive). **Confirmed not a blocker for E.9 acceptance.**

## i18n coverage — current state

```
$ pnpm check-i18n
check-i18n — OK  (en=361 keys, de=361 keys)
```

Both locales carry **361 fully-parallel keys** with placeholder parity verified. The allowlist of intentionally-identical strings (e.g. "CRF", "Audit Trail", "SDV", "MFA", "OID", "Status", "Filter", "Export", "Monitor") is documented in [`scripts/check-i18n-coverage.mjs`](../../../web/src/spa/scripts/check-i18n-coverage.mjs) and references the Bundesinstitut für Arzneimittel und Medizinprodukte's GCP-Inspektionsleitfaden glossary for the German clinical-IT convention.

Default locale on first load is **de-AT** (Austrian German); fallback is **en**. Locale switching from the user's first-login profile completion is wired but the visible language-switcher chip is deferred to a follow-up PR.
